# ISKCON AI Voice Security Alert — Complete Process (End-to-End)

**What it is:** when a temple CCTV camera detects someone crossing a restricted
line, this system **automatically phone-calls every security guard** and speaks a
spoken alert. Guards can press a key to **acknowledge** or **snooze** further
alerts. It runs alongside the existing Hik-Connect app notification (untouched).

- **Live backend (API):** `https://iskcon-temple-aicalling.fly.dev`
- **Live dashboard:** `https://iskcon-temple-aicalling-web.vercel.app`
- **Agent ID:** `da45de76-3e2d-4e2f-b1d5-e2b2cda0daf4`

## 1. THE BIG PICTURE (the whole flow in one diagram)

```
  🎥 Camera detects a line-crossing
        │
        ▼
  📦 NVR (LAN only, 192.168.3.x)
        ├──► Hik-Connect app notification        (EXISTING — untouched)
        │
        └──► 🍓 Raspberry Pi  (app.py, Flask :5050, on temple network)
                   │   checks active-hours + cooldown, then fires ONE request:
                   │
                   │   POST /api/alert
                   │   Authorization: Bearer acai_………   ← the API KEY
                   │   { agentId, fromNumber, phones:[{name,phone}, …] }
                   ▼
             ☁️ Backend on Fly.io  (the "AI platform")
                   │  1. Verifies the API key
                   │  2. Loads the alert agent (message + voice + call type)
                   │  3. Checks snooze window (skip if paused)
                   │  4. Creates a call record per guard
                   │  5. Dials ALL guards at once (no queue) via Plivo
                   ▼
             📞 Plivo places each call ──► guard's phone rings
                   │  On answer → backend returns the spoken message
                   │  (voice generated by 🗣️ Cartesia, cached in MongoDB)
                   │  → guard hears the alert, may press a key (ack / snooze)
                   ▼
             🛡️ All guards are alerted. Outcome saved to Call History.
```

**One line:** *Camera → Pi → `/api/alert` → dial all guards → speak the alert → keypad response → logged.*

## 2. THE API — how a call is triggered (the integration point)

Everything starts with a single API call. This is what the Raspberry Pi (or any
system) sends. **This is the entire integration — one endpoint.**

### `POST /api/alert`

**Auth:** an API key in the header — `Authorization: Bearer acai_<hex>`
(minted in the dashboard → API Keys page; stored only as a hash).

**Request body:**
```json
{
  "agentId":    "da45de76-3e2d-4e2f-b1d5-e2b2cda0daf4",
  "fromNumber": "+9180XXXXXXXX",
  "phones": [
    { "name": "Guard 1", "phone": "+916303639014" },
    { "name": "Guard 2", "phone": "+919542763698" }
  ]
}
```
- `agentId` — which alert message to speak (created in the dashboard).
- `fromNumber` — the Plivo caller ID (optional; falls back to the saved default).
- `phones` — the guards to ring (`string[]` or `[{name,phone}]`). Numbers are
  normalized to E.164; placeholders/duplicates are dropped.

**Response (200):**
```json
{
  "message": "Alert dialing 2 number(s) via plivo.",
  "calls":   [ { "callId": "…", "phone": "+916303639014" }, … ],
  "dialed":  2,
  "skipped": 0
}
```

**cURL example:**
```bash
curl -X POST https://iskcon-temple-aicalling.fly.dev/api/alert \
  -H "Authorization: Bearer acai_XXXXXXXXXXXXXXXXXXXX" \
  -H "Content-Type: application/json" \
  -d '{ "agentId":"da45de76-3e2d-4e2f-b1d5-e2b2cda0daf4",
        "phones":[{"name":"Guard 1","phone":"+916303639014"}] }'
```

> If the alert agent is currently **snoozed**, the response is still 200 but with
> `"suppressed": true` and `"dialed": 0` — no calls are placed until the pause ends.

## 3. WHAT HAPPENS ON THE CALL (per guard)

1. Plivo rings the guard's phone.
2. On answer, the backend serves the **spoken alert message** (rendered once by
   **Cartesia** text-to-speech, then cached in MongoDB → ₹0 on every later call).
3. Depending on the agent's **Call type**, the guard can respond on the keypad:

| Call type | What the guard hears / does |
|-----------|-----------------------------|
| **One-way announcement** | Hears the message, call ends. No keypad. |
| **Announcement + acknowledge** | Hears the message, then *"press any key to confirm."* Press any key → call marked **acknowledged** in history. |
| **Announcement + keypad snooze** | Hears the message, then the options. Press **0** = pause alerts 30 min · **1–9** = pause that many hours. During the pause, new crossings place **no calls**. |

4. Plivo reports the outcome → the call is saved in **Call History**
   (`completed` / `no_answer` / `failed`, with duration + acknowledged flag).

## 4. THE PROVIDERS (what each does)

| Provider | Role | Where the keys live |
|----------|------|---------------------|
| **Plivo** | Places the phone calls (telephony) | Dashboard → Settings (encrypted in DB) |
| **Cartesia** | Turns the alert text into a spoken voice (TTS) | Dashboard → Settings |
| **Gemini** *(optional)* | Helps draft/translate the alert text in the dashboard — **never used during a call** | Dashboard → Settings |
| **MongoDB** | Stores agents, call history, cached audio, users, encrypted credentials | Fly secret `DATABASE_URL` |

**Credentials are entered in the dashboard (Settings)** and stored **encrypted**
in MongoDB (AES-256-GCM). Environment variables act only as a fallback. So the
client can manage provider keys from the UI — no redeploy needed.

## 6. THE RASPBERRY PI (the local bridge)

The NVR can only send its event on the **local network**, so a small always-on
**Raspberry Pi** in the CCTV room catches it and forwards it to the cloud.

- Runs `pi/app.py` (Flask on `:5050`) as a systemd service (auto-start, auto-restart).
- `setup_nvr.py` tells the NVR to POST line-crossing events to the Pi.
- `pi/config.json` holds: the **API key**, **Agent ID**, backend **URL**, caller
  **number**, guard **phone numbers**, active hours, cooldown.
- On each crossing it sends `POST /api/alert` (3× retry so one network blip never
  loses an alert). Active-hours + cooldown are enforced on the Pi.

**Hardware:** Raspberry Pi 4 Model B (4GB) + high-endurance microSD + official
power supply + case with cooling + Ethernet cable, plugged into the CCTV UPS.

## 8. GO-LIVE CHECKLIST

1. **Settings** → enter real **Plivo** (Auth ID + Auth Token + a number you own)
   and **Cartesia** (API key + voice). Confirm `/api/health` shows
   `plivo:true, cartesia:true`.
2. **Alert Message** → set the spoken message, language, **Call type**, voice.
   Note the **Agent ID**.
3. **API Keys** → mint a key for the Pi.
4. **Dashboard → Send a test call** to your phone → confirm it rings + the keypad
   response works.
5. **Raspberry Pi** → fill `pi/config.json` (API key, Agent ID, URL, guard
   numbers), run `pi_setup.sh` (service) + `setup_nvr.py` (point NVR at the Pi).
6. Walk the camera line → all guards get called automatically. **Live.** ✅
